Direct, fanout e topic

O node anterior deixou a exchange com bindings legíveis. Agora a pergunta muda: quando uma publicação chega com uma routing key, a exchange deve comparar por igualdade, copiar para todos os destinos ou testar um padrão?

Pesquisa consultada em 2026-06-08: documentação oficial RabbitMQ 4.3, tutoriais oficiais e especificação AMQP 0-9.

A diferença está na pergunta que a exchange faz

Imagine três publicações pequenas: orders.created, logs.error e audit.user.login. Todas chegam a uma exchange, e todas carregam uma routing key. O que muda entre direct, fanout e topic não é a existência da key; é a pergunta que a exchange faz aos seus bindings.

Direct

Igualdade exata

A key da publicação é comparada com a key registrada no binding. Se os textos são iguais, aquele destino entra na rota.

Fanout

Cópia ampla

A publicação segue para todos os destinos ligados. A routing key pode existir, mas não participa do filtro.

Topic

Padrão por partes

A key é lida como segmentos separados por ponto, e o binding registra um padrão que pode ter curingas.

Esse trio é suficiente para muita topologia cotidiana porque cobre três intenções diferentes: escolher uma etiqueta exata, enviar cópia ampla ou deixar a fila interessada em uma família de keys.

Direct resolve quando a etiqueta precisa bater

Quando a intenção é uma etiqueta exata, a exchange só precisa perguntar se a routing key da publicação é igual à binding key da regra. Esse é o papel da direct exchange.

Publicação

Key enviada

orders.created

A key vem com a mensagem. Ela não escolhe fila por si só; ela será comparada com os bindings da exchange.

Bindings

Regras registradas

Fila de notificação orders.created combina.
Fila de auditoria orders.created combina.
Fila de cancelamento orders.cancelled não combina.
Consequência

Mais de uma saída

A palavra direct não limita a rota a uma fila. Se duas filas têm bindings com a mesma key, as duas recebem cópias da publicação.

Por isso direct é suficiente quando a decisão principal cabe em uma key completa: orders.created, invoice.paid, logs.error. Ele fica fraco quando a key precisa carregar mais de um eixo de escolha e cada destino quer uma fatia diferente desses eixos.

Fanout resolve quando escolher não faz parte da intenção

Às vezes a publicação não precisa selecionar uma categoria. O objetivo é que todos os destinos ligados recebam uma cópia. Nesse caso, a fanout exchange transforma a lista de bindings em uma lista de saídas, sem usar a routing key para filtrar.

Situação Leitura correta Consequência
logs.error chega a uma fanout exchange A key pode estar presente, mas a exchange não a consulta. Todas as filas ligadas recebem cópia.
Uma fila ligada grava histórico e outra atualiza tela São destinos diferentes recebendo a mesma publicação. Isso é broadcast por topologia, não disputa entre consumidores.
Uma fila queria apenas erros críticos Fanout não expressa esse filtro. A topologia precisa de outra regra de decisão.

O ponto de corte. Se todos os destinos ligados devem receber tudo, fanout é direto e legível. Se algum destino precisa escolher por categoria, fanout já não está respondendo à pergunta certa.

Topic entra quando a key vira uma frase curta

Quando uma key exata fica estreita demais, a exchange pode ler a routing key como segmentos separados por ponto. A topic exchange compara essa key com um routing pattern registrado no binding.

Antes de olhar os curingas, veja a forma: audit.user.login tem três segmentos. O primeiro sugere uma família, o segundo uma área, o terceiro uma ação. O binding pode fixar alguns segmentos e deixar outros variarem.

audit

Primeiro segmento. Pode representar a família de eventos que interessa a uma fila.

user

Segundo segmento. Pode variar quando a fila aceita várias áreas dentro da mesma família.

login

Terceiro segmento. Pode ser fixo quando a fila só quer uma ação específica.

# routing key publicada
audit.user.login

# padrões registrados em bindings
audit.*.login => combina com exatamente um segmento no meio
audit.# => combina com zero ou mais segmentos depois de audit
orders.# => não combina com audit.user.login

O asterisco * cobre exatamente um segmento. A cerquilha # cobre zero ou mais segmentos. Essa diferença pequena muda muito a amplitude da rota: audit.*.login é seletivo; audit.# pega uma família inteira; # sozinho fica amplo a ponto de se aproximar de fanout para aquele binding.

Como escolher sem decorar tipo por tipo

A escolha fica mais simples quando começa pela intenção da fila, não pelo nome da exchange. Pergunte qual leitura a fila precisa fazer sobre cada publicação.

Intenção de roteamento Tipo que costuma expressar melhor Leitura da key Cuidado imediato
Receber apenas uma etiqueta exata, como orders.created. Direct Comparar igualdade entre binding key e routing key. Mais de uma fila pode usar a mesma key.
Receber tudo que passar por aquela exchange. Fanout Não usar routing key como filtro. Não usar quando o destino precisa escolher por categoria.
Receber uma família de keys com partes variáveis. Topic Comparar segmentos e curingas. # pode abrir demais a rota.

O risco do padrão amplo. Em topic, # é útil quando a fila aceita uma família inteira de keys. Ele é perigoso quando vira atalho para "qualquer coisa". Quanto mais amplo o padrão, menos a topologia comunica a intenção real de cada destino.

Quando essa leitura está clara, direct, fanout e topic deixam de ser nomes para memorizar. Eles viram três respostas para a mesma pergunta: a publicação deve bater exatamente, ir para todos os destinos ligados ou combinar com um padrão por segmentos?
